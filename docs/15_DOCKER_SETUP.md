# Purpose

This document defines the Docker-first deployment and local development setup for HIPForge. It ensures the application is containerized, reproducible, and easily deployable. Docker Compose provides the necessary abstraction to achieve this by packaging each service into its own container, ensuring dependency isolation and environmental consistency across different machines. The setup prioritizes simplicity for initial deployment while allowing for future scalability.

---

# Core Components

HIPForge utilizes Docker containers for its three main services:

1.  **Backend**: The FastAPI application, responsible for workflow_engine, API handling, and AI agent coordination.
2.  **Frontend**: The Next.js application, providing the user interface.
3.  **Redis**: The in-memory data store for shared state and real-time communication.

---

# `docker-compose.yml` Structure

The `docker-compose.yml` file defines the services, networks, and volumes required to run HIPForge.

```yaml
version: '3.8'

services:
  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    ports:
      - "8000:8000"
    volumes:
      - ./workspace:/app/workspace
    env_file:
      - .env
    depends_on:
      - redis
    networks:
      - hipforge-network

  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile
    ports:
      - "3000:3000"
    env_file:
      - .env
    depends_on:
      - backend
    networks:
      - hipforge-network

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    networks:
      - hipforge-network
    volumes:
      - redis_data:/data

volumes:
  redis_data:

networks:
  hipforge-network:
    driver: bridge
```

# Dockerfiles

## Backend Dockerfile (`backend/Dockerfile`)

This Dockerfile builds the FastAPI backend service.

```dockerfile
# Use an official Python runtime as a parent image
FROM python:3.10-slim-buster

# Set the working directory in the container
WORKDIR /app

# Install system dependencies required for hipcc and hipify-clang
# These will be installed in a separate container or managed via a base image
# For now, assume a base image with ROCm/HIP SDK is used or these are pre-installed
# RUN apt-get update && apt-get install -y --no-install-recommends \
#     build-essential \
#     libhip-dev \
#     hipify-clang \
#     && rm -rf /var/lib/apt/lists/*

# Copy the current directory contents into the container at /app
COPY ./requirements.txt .

# Install any needed packages specified in requirements.txt
RUN pip install --no-cache-dir -r requirements.txt

# Copy the rest of the application code
COPY . .

# Expose port 8000 for FastAPI
EXPOSE 8000

# Run the FastAPI application
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

**Note**: The installation of `libhip-dev` and `hipify-clang` is commented out. In a production environment, a specialized base image containing the ROCm/HIP SDK would be used, or these dependencies would be managed in a separate compiler container. For the hackathon, it's assumed the host environment or a pre-configured base image provides these.

## Frontend Dockerfile (`frontend/Dockerfile`)

This Dockerfile builds the Next.js frontend service.

```dockerfile
# Use an official Node.js runtime as a parent image
FROM node:18-alpine

# Set the working directory in the container
WORKDIR /app

# Copy package.json and package-lock.json (if available)
COPY package*.json ./

# Install dependencies
RUN npm install

# Copy the rest of the application code
COPY . .

# Build the Next.js application
RUN npm run build

# Expose port 3000 for the Next.js app
EXPOSE 3000

# Start the Next.js application
CMD ["npm", "start"]
```

# Volumes

- **`./workspace:/app/workspace` (Backend)**: This volume mounts the host's `./workspace` directory into the backend container at `/app/workspace`. This is crucial for persisting migration-specific files, logs, and reports generated by the Workflow Engine and AI agents. This ensures that data is not lost when the container is stopped or removed.
- **`redis_data:/data` (Redis)**: A named Docker volume for Redis to persist its data. This prevents Redis data from being lost if the Redis container is restarted.

---

# Networking

A custom bridge network, `hipforge-network`, is defined to enable seamless communication between the `backend`, `frontend`, and `redis` services. This isolates the application's internal traffic from the host network and simplifies service discovery using service names as hostnames.

---

# Environment Variables

Environment variables are loaded from a `.env` file at the root of the project and passed to both the backend and frontend services. This allows for flexible configuration without hardcoding sensitive information or changing Dockerfiles.

Example `.env` file:

```
REDIS_URL=redis://redis:6379
FIREWORKS_API_KEY=your_fireworks_api_key
WORKSPACE_PATH=/app/workspace
DEFAULT_RETRY_BUDGET=5
WORKSPACE_SIZE_LIMIT=100MB
LOG_LEVEL=INFO
NEXT_PUBLIC_BACKEND_URL=http://localhost:8000
```

# Build and Runtime

## Building Images

To build the Docker images for all services:

```bash
docker-compose build
```

## Starting the Application

To start all services in detached mode:

```bash
docker-compose up -d
```

## Stopping the Application

To stop and remove all services, networks, and volumes:

```bash
docker-compose down -v
```
